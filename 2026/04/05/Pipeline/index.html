Single-Cycle Processor

  • 定义：在Single-Cycle processor中，每条指令都在一个时钟周期（Clock Cycle）内完成
    • 即：IF(instruction fetch), ID(instruction decode), EX(execute), MEM(memory access), WB(write back)，都在同一个cycle内完成
    • 所以： 一个周期必须足够长，以至于它可以完成最慢指令。
  • 优点：
    • 控制逻辑相对简单（一个时钟周期内只完成一条指令）
  • 缺点：
    • 硬件资源需求大
    • 简单指令也被迫占用最慢指令的执行时间（$e.g. add和load使用同样长的时钟周期$）

Multiple-Cycle Processor

  • 定义：在Multiple-Cycle Processor中，一条指令分成多个时钟周期完成
    • 即： 不同阶段分散到不同cycle中执行，如：
      • 第一个cycle：IF
      • 第二个cycle：ID
      • 第三个cycle：EX
      • 第四个cycle：MEM
      • 第五个cycle：WB
  • 相比于SC Processor的优化：
    • 不同指令可以使用不同数量的周期
    • 每个周期可以被设计地更短，因为一个周期只用完成一个指令的一部分（如IF,ID…）
    • 避免了Single-Cycle中“所有指令都受最慢指令拖累”的问题
  • 仍然存在的不足：
    • 整个处理器仍然是顺序执行指令，在极大规模的指令数量下，优化程度仍然不够看

Pipelined Processor

  • 定义：在Pipelined Processor中，指令仍然分成多个阶段执行，但是不同指令的不同阶段可以同时执行。
    • 如在同一时刻可能存在：
      • 指令1在EX
      • 指令2在ID
      • 指令3在IF
  • 核心优化逻辑： pipeline不是把一条指令压缩得更短，而是让多条指令重叠执行，从而提高throughput

Pipeline Hazards

  • 值得注意的是，即使pipeline看起来极大程度的提高了指令执行的效率（以指令的阶段数为参数），这样重叠执行的方式也带来了三种危害，这些危害使得后续的指令执行在ID阶段就出现STALL的error

Structure Hazards

  • 由于硬件资源数量不足，导致多条指令在同一时刻竞争同一资源，而导致的危害
  • 如：
    • 在Von Neumann结构中，指令和数据共用同一个memory，所以在某cycle中可能出现：指令I1在MEM阶段，要访问内存中的数据；指令I4在IF阶段，要从内存里取下一条指令；两者同时都要用memory，导致冲突发生
    • 若处理器只有一个division functional unit，且一次除法要跑很多周期。如果有两条连续的除法指令，那么在第一条指令还在EX的时候，第二条指令也到了EX的阶段，这时候就出现了第二条指令的stall，也属于structure hazards
  • 值得注意的是，在现代scalar processor（标量体系结构，每个周期主要发射一条指令）中，structure hazards并非常见，因为大部分寄存器通常有足够的端口，且ALU一般执行地很快（除了除法单元）或者也为流水化的实现逻辑。
  • 同时，避免scalar processor的方法在现代处理器中也得到大规模的应用
    • 把instruction memory和data memory分开放（Harvard Machine），避免访问冲突
    • 放置两个加法器，因为“增加PC”和“进行算数运算”往往都需要加法，使用两个加法器可以避免structure hazards
    • 重新安排指令的顺序，不让两个很可能会重复使用同一寄存器/运算单元的指令连续出现（如在两条div指令之间重排入几条不需要divider的指令）
    • 重新设计ISA，尽量避免抢处理器的指令出现
  • 由于“新指令必须等待老指令使用完它占用的register”而产生的instruction stall，称为pipeline bubbles

从Structure Hazards思考指令多阶段的必要性

  • 考虑一个R-type指令，如add x1 x2 x3，它经历了IF,ID,EX,WB，但是并不需要访问data memory，也就说不必执行MEM阶段。那么是否可以在处理R-type指令时，直接跳过EX，提前写回寄存器呢
  • 显然是不行的。
  • pipeline中，多指令重叠执行的一个特点就是，不同指令不会同时执行同一阶段，若在一些特定指令中跳过某些阶段的执行，那么很可能出现不同指令同时执行某阶段——如WB——的情况，这种时候就会出现硬件资源冲突，产生structure hazard

Data Hazards

  • 由于后面的指令想用数据，但是该值依赖于前面的指令，且尚未准备好，而造成的危害
    • Data hazards arise from data dependencies
  • 例：
    • add x9 x0 x1 该指令中x9的值是在EX阶段结束时就绪，在WB阶段写回x9寄存器的
    • sub x10 x9 x3 该指令的ID阶段就需要读取寄存器x9中的值，但是显然这时候x9的值还没有被上一条指令写回寄存器
    • 在该例子中，pipeline被stall，直到上一条指令的WB阶段，即整条指令完成。分析该过程，在第一条指令执行EX,MEM,WB时，第二条指令都卡在ID，即产生了三个pipeline bubbles
  • 该例中是“在当前cycle中，尝试读未来cycle中得到的数据”，不妨考虑新情况“在当前cycle中，尝试读取当前cycle中，由其它指令的某阶段得到的数据”，如下图

  • 如图中所述，为解决这种冲突，对于每一个寄存器在同一个cycle中的行为，将其设计为“先写后读”；那么同一个周期里就能读到新值
  • 注意： 这种处理方式在同一个cycle中重复使用了某个寄存器，但是这并不存在structural hazards，因为每个寄存器有分开的端口，支持同时进行读/写

Data Hazards的解决方法

  • Solution 1：Pipeline Stall at ID stage (Interlocking)
    • 最简单的解决方法就是人为地构造stall，在读不到对应数据的ID阶段停住，直到能读到对应数据；这样就解决了由于引用旧数据而造成的运算结果错误
    • 这种解决方法需要在硬件上添加相关的控制电路，实现对指令依赖的检测、指令执行的暂停与继续
    • 优点：简单可靠、容易理解、不需要复杂的数据转发路径
    • 缺点：浪费周期，降低吞吐量，CPI变大，性能被拖累
  • Solution 2：Data Forwarding (Bypassing)
  • 如图所述，forwarding的本质是在结果一被算出来时就使用它，不必等到WB写回寄存器
  • 而为了实现这种“转发过程”，需要在datapath里加上额外的连线，也要添加forwarding相应的硬件控制
  • 上图中是add类指令(R-type)的forwarding，它们的运算结果在EX阶段就得出；但lw类指令（I-type）的forwarding是从MEM开始而非EX，因为在EX阶段它们得到的只是寄存器地址，MEM阶段才完成了值的取得，如下图所示。

思考：为什么forwarding的终点是EX，但是指令STALL却是停在ID？

  • forwarding的终点是EX： 读取寄存器发生在ID，但是真正使用数据在EX，所以在进入EX的edge处通过forwarding传入正确的新值，ALU仍然可以得出正确的结果
  • STALL停在ID： hazard detection发生在ID，当一条指令走到ID时，逻辑电路检查：
    • 这条指令要读那些reg
    • 前面pipeline的指令里会不会WB这些reg
    • 这些值到它将来进入EX时，来不来得及通过forwarding或正常WB得到正确的新值
      • 若来得及就继续进入EX
      • 若来不及就STALL在ID

检测Data Hazard

  • 讨论检测Data Hazard前，我们先回到Data Hazard的特性：“后面的指令想用数据，但是该值依赖于前面的指令”，那么自然，检测Data Hazard的方法就落到了“比较寄存器”上。
  • 也就是说，如果前面某条指令将要写回的目标寄存器，刚好等于当前这条指令要读取的源寄存器，那就说明有data dependency，可能需要forwarding
  • 回顾前文，forwarding有两种情况
    • EX->EX
    • MEM->EX
  • 所以在比较寄存器时，如下四种情况可能出现Data Hazard：
    • 1a. EX/MEM.RegRd = ID/EX.RegRs1
    • 1b.EX/MEM.RegRd = ID/EX.RegRs2
    • 2a.MEM/WB.RegRd = ID/EX.RegRs1
    • 2b.MEM/WB.RegRd = ID/EX.RegRs2
    • 值得一提的是，此处的register正是前文提到的pipeline register
      但是这些情况并不一定出现Data Hazard，需再考虑一下情况：
  • 1.若前面的指令不写入寄存器，那就不用forward（如 beq x1 x2 L）
  • 2.若目标寄存器是x0，那么也不用forward，因为在RISC-V中，x0恒等于0，对其的写入会被忽略

Solution 3: Compiler Code Transformations

  • MIPS(Micro-Processor without Interlocked Pipe Stages)是不存在Interlock硬件的系统。它在处理Data Hazard时，编译器重排指令，以达到无冲突的需求。
  • 若在“产生结果”和“使用结果”的间隔之间，没有合适的可插入指令，那么就插入NOP
  • 对指令的重排是相当有价值的，能对执行进行大幅度的优化；可以达到 “能不stall就不stall” 的编译顺序。这和Solution 1结合起来就是目前编译器处理hazard的逻辑之一：编译器做前置优化，硬件最后完成必要的stall
  • 但是注意，重排并不是无条件，只要register字段不匹配就可以随意重排的：
    • 1.Memory dependences unknown：涉及内存访问时，由于比较寄存器编号并不能比较该指令所对应的内存地址，所以不能判断交换它们会不会访问同一地址、会不会改变语义
    • 2.Branches，Indirect Branches, call/ret：控制流边界会大幅度地影响指令的执行顺序和范围，所以当编译器尝试跨过这些控制流边界来进行指令顺序的重排时，很可能会改变原有的语义
  • Memory Dependences如下图：
  • 我们不能确定0($pr5)和c是不是同一个地址，所以不能确定指令7/8能不能交换
  • 大部分情况下，即使两个指令中的地址很可能不是同一个地址，编译器也不会交换它们的执行顺序，因为编译器是conservative的

用拓扑排序求指令reorder后的执行顺序

  • 不妨考虑Data Hazard中的顺序关系，我们在重排指令顺序时，必须满足若干条“指令A必须在指令B之前执行”的约束条件；在这种情况下我们从节点A向节点B连一条有向边，表示A必须在B之前执行

  • 分析完所有指令间的关系并构建相应的节点和有向边后，我们就能得到一张有向图；不妨考虑该有向图的一个拓扑序，正是满足节点间先后关系的一个排列。也就是说，只要一个顺序满足DAG里的所有依赖约束，它就是一个legal schedule。

  • 所有的拓扑序都对应一个合法的指令顺序，Code Scheduling就是在所有拓扑序中找到一个“最好”的

    • “最好”可能是总周期最少，stall最少，资源利用更好，更适合某个具体处理器

  • 但是寻找最好的拓扑序的过程并不能通过简单地遍历每一个拓扑序来实现——合法排列的数量太多了，以指数级增长

  • 为寻找最合适的拓扑序，我们给DAG再做细化，为每条边加上权重，指代这条依赖所对应的延迟；即若该边为从A指向B，那么权重的意味是：在A产出结果后，B最早要等多久才能安全开始

  • 那么我们仍然需要一个判断拓扑序列优质程度的指标：这里引入critical path，指令依赖图中，按执行时间（或周期数）度量的最长依赖路径

  • 注意： 最长依赖路径决定了该段代码的优化上限——这段代码再怎么优化，最快也不可能比这条路径更短

  • 而在构建一个合法，且最优的拓扑序时，我们采用贪心的策略，尽可能早地调度关键路径上的点，这是因为作为优化的上限，若该路径上的某条指令又被拖延执行，那么整条链只会更长，完成时间更久，优化效率更低

  • 接下来叙述寻找critical path的方法：

    • 从叶子节点（出度为0）往根节点（入度为0）遍历DAG，给每个节点标上值，表示从这个节点开始，到这个basic block结束，可能产生的最大延迟（MD），如下图，更新方式与dijkstra相似，以此可以求得本图中的critical path是1->3->4->6->7

  • MD与scheduling的关系：MD大的节点，通常更靠近关键路径，即MD可作为一个priority指标

  • List Scheduling 流程（basic block scheduling）：

    • Step 1: Build Dependence DAG，节点指代指令，连边指代存在依赖关系，边权指代latency
    • Step 2: Mark each node with maximum delay to the end of block，求出前文所述的MD，作为后续遍历时的priority参考
    • Step 3: Traverse the DAG from the root toward the leaves, selecting nodes from a candidate list to schedule. Keep tracking of the current cycle to minimize possible stalls，这里的关键是从候选列表中挑选下一个调度的节点
      • candidate list：当前已经ready、可以被调度的集合，编译器在每次选择下一条执行的指令时，都是在candidate list中选择
      • 减少stall的思路：优先调度candidate list中priority（MD）最高的节点
    • candidate list中同时有好几条指令都能发时，编译器优先选择的逻辑：（heuristic）
      • 1.优先选择那些现在发出去不会导致stall的节点
      • 2.若前面条件相同，则选择到basic block结尾最大延迟（MD）更大的节点
      • 3.若前面条件相同，则
        • a.优先保持原始顺序
        • b.优先选择children较多的节点：children较多的节点意味着后续有更多指令依赖于这条指令，可以理解为这条指令在整个block中较为重要
        • c.优先选择那些是某个寄存器最后一次使用的节点：更早释放某些寄存器，降低live value数量，减少后续register allocation的压力
    • 注意：heuristic是一种局部贪心，不能在数学上保证“全局最优”，但是是在大多数情况下，比较有效、成本也低的经验策略。毕竟计算机的效率够高，我们不必求出严格的最优解，只要快速做出“够好”的选择就行。

Control Hazard

  • 由于程序里出现了改变PC走向的指令，导致pipeline无法正常延续，产生stall的情况
    • 理解：直到brach指令的跳转条件被判断或jal指令的PC+immed数被计算出来，我们都不能确定下一条指令是否该在下一条被执行
    • 也就相当于，下一条指令的IF会一直stall（编译器不确定要fetch哪条指令），直到control指令被处理完毕
    • 注意： 相应的，Branch指令的结果在EXE/MEM得出
  • 不难想到，最简单的处理方法就是以时间换正确性，即在每个Branch/Jump指令处都插入bubbles，强制等EXE执行完再进行下一条正确指令的IF
    • 在这种解决方式下，假设20%的指令是branch/jump指令，且忽略pipeline的fill和drain造成的额外花费
    • 则average CPI从1变为$0.23+0.81=1.4$
  • 若我们假设每次都是不跳转，即直接执行下一条指令，那么
    • 若branch实际上要taken，那么会flus掉两条指令（2-cycle penalty）
    • 不妨做一个性能估算
      • 20%指令为branch
      • 50%的branch指令最后需要taken
      • average CPI = 1
      • 则Penalty = $2(flushed) * 0.2 *0.5 = 0.2 CPI$,即会出现20%的penalty

Static Branch Prediction

  • 在编译阶段就决定好prediction的模式，称为Static Branch Prediction
  • 永远预测不taken和永远预测taken就属于static branch prediction
  • 常用的两个预测规则：
    • Predict a backward branch as taken：回跳的branch，预测为taken，因为它通常对应loop back，如执行完一次循环体后跳回先前的循环条件判断，而loop back除了最后一次之外都是要taken的
    • Predict a forward branch as fall-through，向前跳的分支，预测成走顺序路径(fall-through)，即预测为不taken（该规则的原理是编译器有时会把更值得执行的那段放在顺序路径上）
  • 可以由硬件实现支持
    • 如有些ISA在branch instruction里留了一个prediction bit来指示硬件这条branch更可能taken还是not taken

Dynamic Branch Prediction

  • 在程序执行过程中实时做出的，CPU会观察一条branch过去是
    • 常taken
    • 常not taken
    • 还是来回变
  • 根据这个记忆化过程，来预测它下一次会怎么样
  • Branch History Table(BHT)：
    • 用branch instruction’s address来作为索引完成表格
    • 存储branch outcomes(taken/not taken)
    • 在执行branch时根据BHT来决定是否跳转，能大幅度减少flush的次数

Branch Prediction at IF Stage

  • 值得注意的是，Branch指令需要在IF阶段就做出对下一条指令的预测，不然下一条指令仍然会卡在IF
  • 但是在IF阶段我们只是取了该条指令的PC，理论上在ID才能解码出这条指令的类型
  • 为了不产生stall，在IF阶段我们就直接用该条指令的address去BHT里检查
    • 若该表项没有有效历史信息，那么默认预测 NOT Taken，这也与“若该条指令不是branch指令”的实际结果（直接执行下一条）相符
    • 若该表项有历史信息，按照历史信息执行
  • 执行完branch指令后，再根据指令的执行结果去更新BHT表格

Basic Predictor

  • 1-bit Predictor：
    • 预测结果与上次的实际结果相同
    • 在多重循环里经常出错
  • 2-bit Predictor :
    • FSM(Finite State Machine)
    • 存在四种state
      • 11 Strongly Taken
      • 10 Weakly Taken
      • 01 Weakly Not Taken
      • 00 Strongly Not Taken
    • Taken则给状态+1
    • Not Taken则给状态-1
    • 10或11则taken，00或01则not taken
  • Correlated Prediction
    • 不只看这条指令历史的执行结果，还检查最近几条影响控制流状态的branch结果
    • 称作2-level adaptive
      • level-1：记录最近的branch outcome pattern
      • level-2：根据pattern查表里的预测状态
      • adaptive :随着运行更新pattern和table
    • Shift Register(Global Branch History Register，GHB):
      • 每个bit记录一条指令是否taken
      • 新的指令执行时，就pop掉最旧的指令，类似queue
    • Pattern History Table(PHT):
      • 每个表项是一个2-bit predictor counter
      • 用GHB的内容来索引PHT，以此作为指令执行的依据
      • 当前branch执行完后，用更新2-bit Predictor的方式更新表项
    • 对Correlated Prediction的理解： 以“控制流中先前状态”为关键字，统计控制流中先前的执行结果来作为当前执行的依据

Prediction的局限性

  • 当我们讨论Branch指令时，我们讨论的实际上是两个组成部分
    • 是否跳
    • 跳到哪
  • Prediction只解决了“是否跳”的问题，而“跳到哪”需要在EX阶段计算出
  • 所以对于那些taken branch来说，仍然会存在1-cycle or 2-cycle的penalty（看address是在ID还是在EXE计算）
  • 解决方案：Branch Target Buffer
    • 对目标地址进行缓存
    • 能够支持缓存的原理：多次执行一个branch指令时，它的immediate通常是不会被改变的
    • 缓存方式：在IF阶段取到某条instruction时，用它的PC去查BTB
